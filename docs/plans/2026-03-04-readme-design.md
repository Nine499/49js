# README 重构设计（多脚本目录式）

日期：2026-03-04
范围：`README.md`（用户安装导向，覆盖多个 userscript）

## 1. 背景与目标

当前仓库已包含多个脚本（`pbs.twimg.com-orig.js`、`fanbox-Kemono.js`），但 README 仍以单脚本叙述为主，不利于用户快速判断“装哪个脚本、在哪生效、如何安装与排障”。

本次目标：

- 将 README 重构为“多脚本目录式”入口文档
- 面向用户安装与使用，强调可直接执行的步骤
- 提供“远程安装（推荐）+ 本地安装（开发）”双路径
- 为每个脚本提供一致的信息模板，降低认知成本

非目标：

- 不改动脚本业务逻辑
- 不新增额外文档站点或自动化发布流程

## 2. 受众与成功标准

受众：

- 主要：需要快速安装 userscript 的终端用户
- 次要：在本地调试脚本的开发者

成功标准：

1. 用户在 10 秒内可从目录表判断脚本选择
2. 用户可直接点击远程安装链接完成安装
3. 用户可通过 FAQ 完成基础排障
4. README 内容与当前脚本元数据（`@match`/`@updateURL`/`@downloadURL`）一致

## 3. 方案对比

### 方案 A（推荐）：单 README 多脚本总览

结构：项目简介 → 快速安装 → 脚本目录对比 → 分脚本详情 → FAQ → 版本更新 → License

优点：

- 单入口完整覆盖安装与使用路径
- 与“彻底更新 README”目标一致
- 适配后续新增脚本的扩展

代价：

- README 长度增加

### 方案 B：README 极简 + 分散到其他文档

优点：首页短。
缺点：跳转成本高，不满足本次“彻底更新 README”诉求。

### 方案 C：纯场景化导览

优点：新手友好。
缺点：维护成本高，对多脚本并行扩展不稳定。

## 4. 最终设计

### 4.1 信息架构

1. 项目简介
2. 快速安装
   - 安装 Tampermonkey
   - 安装 `pbs.twimg.com-orig.js`（远程/本地）
   - 安装 `fanbox-Kemono.js`（远程/本地）
3. 脚本目录（对比表）
4. 脚本详情
   - `pbs.twimg.com-orig.js`
   - `fanbox-Kemono.js`
5. FAQ
6. 版本与更新
7. License

### 4.2 写作与风格约束

- 仅中文；技术术语保留 English
- 以可验证行为描述替代宣传性措辞
- 每个脚本详情采用统一模板：作用 / 生效范围 / 行为规则 / 示例 / 注意事项

### 4.3 双安装策略

每个脚本同时提供：

- 远程安装（推荐）：`https://github.com/Nine499/49js/raw/refs/heads/master/<script>.js`
- 本地安装（开发）：仓库中对应脚本文件

## 5. 脚本级内容映射

### 5.1 `pbs.twimg.com-orig.js`

- 生效范围：`*://pbs.twimg.com/*`
- 行为：将 URL 参数 `name` 规范为 `orig`，并在必要时执行 `location.replace`
- 非生效范围：非 `pbs.twimg.com`

### 5.2 `fanbox-Kemono.js`

- 生效范围：`https://www.fanbox.cc/@*/posts/*`
- 行为：通过 Fanbox API 获取数字 `userId`（会话级缓存），注入按钮并跳转至 Kemono 对应 post
- 注意：依赖目标页面可正常访问 API

## 6. 测试与验收

手动验收：

1. README 中全部安装链接可点击
2. 远程链接与脚本文件名一致
3. 目录表信息与脚本元数据一致
4. FAQ 至少覆盖“未生效排查、更新机制、本地与远程区别”

## 7. 变更影响

- 仅文档层变更（`README.md`）
- 不影响脚本运行逻辑
